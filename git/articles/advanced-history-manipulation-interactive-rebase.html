Advanced History Manipulation (Interactive Rebase)

Learning Objectives

  • Understand the purpose and workflow of Interactive Rebase in Git.
  • Master core rebase instructions: pick, reword, edit, squash, and drop.
  • Combine multiple minor commits into a clean snapshot using squashing.
  • Learn how to break apart a single large commit into smaller, atomic commits.
  • Apply clean history manipulation best practices before pushing code upstream.

1. Introduction

As you develop a feature, you may create several small commits for testing, bug fixes, or corrections. Before sharing your work, Git allows you to rewrite local commit history using Interactive Rebase. This helps create a clean and meaningful commit history by combining, editing, reordering, or removing commits.

Note: Interactive Rebase should strictly be used on local commits that have not been pushed to a shared repository.

2. What is Interactive Rebase?

Interactive Rebase allows you to open an editor session to modify, reorder, combine, or discard recent commits before they become part of the main project history.

To launch an interactive rebase session targeting the last 3 commits:

git rebase -i HEAD~3

Git will open your configured text editor showing a list of targeted commits preceded by command keywords.

3. Interactive Rebase Commands

Instruction Purpose & Action
pick Keep the commit as-is in the history.
reword Keep the commit contents, but pause to edit the commit message.
edit Pause the rebase sequence to modify the commit contents or split it.
squash Meld the commit into the previous commit, combining their log messages.
drop Completely remove the commit from the branch history.

4. Squashing, Rewording & Dropping Commits

Squashing Commits

Combine multiple small fixup commits into a single cohesive feature commit:

Commit A → Commit B → Commit C → Commit D
↓ (git rebase -i & squash)
Commit A → Commit B → Commit D'

Rewording Commit Messages

To update an old commit message, change pick to reword next to the targeted commit during the rebase session. Git will pause execution and prompt you for a new message.

Dropping Commits

To permanently delete an unwanted commit from history, change pick to drop (or simply delete the line) during the interactive rebase session.

5. Splitting a Single Commit

If a single commit contains multiple unrelated changes, you can split it into separate atomic commits using this step-by-step workflow:

1. Launch git rebase -i HEAD~n
↓
2. Mark the target commit as edit
↓
3. Unstage the commit: git reset HEAD^
↓
4. Stage & commit individual smaller changes
↓
5. Complete rebase: git rebase --continue

6. Key Commands Reference

Command Purpose
git rebase -i HEAD~n Start an interactive rebase for the last n commits
git rebase --continue Resume the rebase sequence after completing an edit or split step
git rebase --abort Safely cancel the active interactive rebase and restore original state
git reset HEAD^ Unstage the active commit when splitting a commit during edit mode

7. Industry Best Practices

  • Use interactive rebase locally to clean up commit history before opening pull requests.
  • Squash minor fixup or WIP (Work In Progress) commits into clean, descriptive feature commits.
  • Write clear, imperative commit messages across your cleaned timeline.
  • Never rewrite history that has already been pushed to shared team branches.
  • Always test your application suite thoroughly after completing a rebase.

Summary

In this lesson, you learned how Interactive Rebase helps rewrite local commit history before sharing your work. You explored how to squash, reword, drop, and split commits using git rebase -i. These techniques produce a clean, organized commit history, making projects easier to review and maintain.
