Merging & Basic Conflict Resolution

Learning Objectives

  • Understand how Git integrates changes from different development branches.
  • Differentiate between Fast-Forward merges and Three-Way merges.
  • Master executing merges safely into target branches.
  • Identify the core causes of merge conflicts and read conflict markers.
  • Follow a step-by-step workflow to resolve conflicts and commit clean merges.

1. Introduction

After developing features on separate branches, those changes need to be combined into the main project. Git provides merging to integrate branches while preserving project history. Sometimes, changes made in different branches conflict with each other, requiring manual resolution. In this lesson, you'll learn how to merge branches and resolve common merge conflicts.

2. What is Merging?

Merging combines the commits and changes from a feature branch into a target branch (such as main), unifying independent streams of work.

Feature Branch (Commits D → E)
↓ (git merge)
Target Branch (Commit C)
↓
Integrated Timeline / Merge Commit (M)

3. Types of Merges

1. Fast-Forward Merge

Occurs when the target branch has no new commits added after the feature branch was created. Git simply moves the branch pointer forward to the latest commit of the feature branch without creating a separate merge commit.

2. Three-Way Merge

Occurs when both the target branch and the feature branch have diverged with new, independent commits. Git evaluates a common ancestor commit alongside the tip of both branches to construct a new Merge Commit combining both histories.

4. Executing a Merge

To integrate changes from a feature branch into your target branch, follow this process:

First, switch to your destination target branch:

git switch main

Execute the merge command specifying the source branch:

git merge feature-login

Git will automatically evaluate history and perform either a Fast-Forward or Three-Way merge.

5. Understanding Merge Conflicts

A merge conflict occurs when Git cannot automatically reconcile differences between competing changes—such as when the exact same line of a file is modified differently across both branches.

Conflict Marker Anatomy

When a conflict happens, Git inserts conflict marker indicators directly into the affected files:

<<<<<<< HEAD
Hello World (Your current branch changes)
=======
Hello Git (Incoming feature branch changes)
>>>>>>> feature-login

6. Resolving Merge Conflicts Step-by-Step

Follow this structured flow to resolve conflicts cleanly:

1. Run git status to find conflicted files
↓
2. Open files & edit conflict markers manually
↓
3. Stage resolved file: git add <file>
↓
4. Complete merge: git commit

7. Key Commands Reference

Command Purpose
git merge <branch> Merge specified branch into the current branch
git status View unresolved merge conflicts and state
git add <file> Stage files after manually resolving conflict markers
git commit Complete the merge commit after staging resolved files

8. Industry Best Practices

  • Pull the latest upstream changes before initiating a merge.
  • Merge feature branches frequently to keep divergence minimal and reduce conflict size.
  • Keep individual commits small, focused, and single-purpose.
  • Inspect conflict markers carefully rather than blindly deleting surrounding code.
  • Run your test suite immediately following every merge to verify code integrity.

Summary

In this lesson, you learned how Git combines branches using Fast-Forward and Three-Way merges. You also explored how merge conflicts occur when the same code is modified in multiple branches and learned the steps to resolve those conflicts using git status, git add, and git commit. Mastering merging is essential for effective collaboration and maintaining a clean project history.
