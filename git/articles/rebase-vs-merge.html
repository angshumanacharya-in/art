Rebase vs Merge

Learning Objectives

  • Understand the architectural differences between merging and rebasing branches.
  • Analyze how rebasing rewrites commit history to create a clean, linear timeline.
  • Master executing a standard branch rebase step-by-step.
  • Resolve conflicts encountered during a rebase sequence using --continue or --abort.
  • Comprehend and follow the Golden Rule of Rebasing to protect shared team branches.

1. Introduction

Both Merge and Rebase are used to integrate changes from one branch into another. While they achieve the same goal, they do so differently. Merge preserves the complete project history, whereas Rebase rewrites commit history to create a cleaner, linear timeline. Understanding when to use each approach is an important Git skill.

2. Merge vs. Rebase Concepts

Git Merge

Merge combines two branches by preserving all historical commits and creating a dedicated Merge Commit (M) at the point of integration.

Diverged Branches (main vs feature)
↓ (git merge)
Commit M (Merge Commit Created)
↓
Complete Non-Linear History Preserved

Git Rebase

Rebase moves or "replays" feature branch commits directly on top of the tip of the target branch, rewriting the individual commit hashes (D', E') to produce a perfectly linear history.

Base Branch (Commit C)
↓ (git rebase main)
Feature Commits Replayed (D' → E')
↓
Clean & Linear Commit Graph

Why Use Rebase?

  • Produces a clean, highly readable, and linear commit timeline.
  • Avoids cluttering the project graph with repetitive merge commits.
  • Simplifies code reviews, history traversal, and bisecting bugs.

3. Performing a Rebase

To rebase your current feature branch onto the latest tip of another branch (e.g., main):

First, switch to your local feature branch:

git switch feature-login

Execute the rebase command targeting the upstream base branch:

git rebase main

Git will temporarily unapply your feature commits, update your branch pointer to the tip of main, and re-apply each of your feature commits sequentially.

4. Resolving Rebase Conflicts

Because Git re-applies commits one by one during a rebase, conflicts are evaluated for each individual commit in the sequence.

Rebase Conflict Workflow

1. Conflict occurs during commit replay
↓
2. Resolve conflicting files manually
↓
3. Stage resolved files: git add <file>
↓
4. Continue rebase: git rebase --continue

If you run into issues or want to cancel the rebase operation entirely and revert back to your original state:

git rebase --abort

5. The Golden Rule of Rebase

Never rebase commits that have already been pushed to a shared (public) branch.

Rebasing rewrites commit SHA hashes. If you rebase commits that other team members have already pulled, you will create divergent histories and severe merge conflicts for your entire team.

6. Merge vs. Rebase Comparison

Feature Git Merge Git Rebase
History Structure Preserved (True historical record) Rewritten (Clean linear record)
Merge Commits Yes (Creates a merge commit) No (No extra merge commits)
Timeline Non-linear / Forked Linear
Safe for Public Branches Yes (Safe for shared remote branches) No (Only use on unpushed local branches)

7. Key Commands Reference

Command Purpose
git merge <branch> Integrate specified branch into current branch via merge commit
git rebase <branch> Replay current branch commits onto the tip of specified branch
git rebase --continue Resume the rebase sequence after resolving a step's conflict
git rebase --abort Cancel the active rebase operation and restore original state

8. Industry Best Practices

  • Use Merge to integrate public feature branches and preserve full historical audit trails.
  • Use Rebase locally to clean up and update your feature branch before submitting a pull request.
  • Never rebase branches that other developers are actively using or basing their work on.
  • Resolve rebase conflicts carefully step-by-step rather than rushing through commits.
  • Run your test suite immediately following a completed rebase to confirm application functionality.

Summary

In this lesson, you learned the differences between Merge and Rebase. While merge preserves the complete branch history using a merge commit, rebase creates a clean, linear history by replaying commits on top of another branch. You also learned how to perform a rebase, resolve conflicts, and follow the golden rule of never rebasing public branches.
