Branching Fundamentals

Learning Objectives

  • Understand what branches are and how Git handles them internally.
  • Identify the core reasons for isolated feature development.
  • Master creating, listing, and deleting local Git branches.
  • Learn how to switch contexts safely using git switch and git checkout.
  • Follow a standard branch lifecycle from creation to cleanup.

1. Introduction

Branches are one of Git's most powerful features. They allow developers to work on new features, bug fixes, or experiments without affecting the main codebase. By creating separate branches, multiple developers can work independently and merge their changes when they are ready.

2. What is a Branch?

A branch is a lightweight pointer to a specific commit object. Rather than copying files, it simply represents an independent, isolated line of development within your repository tree.

When you create a new branch, development continues separately without altering the primary lineage:

main (Commit C)
↓ (Branch Created)
feature-login (Commit D)

Why Use Branches?

  • Develop new features safely without disrupting production code.
  • Fix emergency bugs quickly without introducing half-finished feature code.
  • Experiment freely with new ideas without consequence.
  • Enable multiple developers to work simultaneously in parallel environments.
  • Keep the main branch clean, stable, and deployable at all times.

3. Creating & Listing Branches

Creating a Branch

To create a new branch pointer at your current commit state without switching to it immediately:

git branch feature-login

Listing Local Branches

View all existing local branches in your repository:

git branch

Your currently checked-out branch is indicated with an asterisk (*):

* main
  feature-login

4. Switching Between Branches

Switch your active working environment to an existing branch:

git switch feature-login

Alternatively, you can use the older syntax:

git checkout feature-login

Note: The git switch command is the modern, dedicated command introduced to separate branch switching from file restoration.

Creating and Switching in One Step

To create a brand new branch and switch to it in a single operation:

git switch -c feature-login

Or using the legacy syntax:

git checkout -b feature-login

5. Deleting Branches

Once a feature branch's work has been safely merged into main, clean up the branch pointer:

git branch -d feature-login

Force Deleting an Unmerged Branch

If a branch contains experimental work that you want to completely discard without merging:

git branch -D feature-login

6. Standard Branch Workflow

The operational sequence for feature development follows a structured loop:

main Branch
↓ (Create Branch)
feature-login Branch
↓ (Make & Commit Changes)
Merge Back to main
↓ (Delete Feature Branch)
Clean Repository History

7. Key Commands Reference

Command Purpose
git branch List all local branches
git branch <name> Create a new branch
git switch <name> Switch workspace to an existing branch
git switch -c <name> Create a new branch and switch to it immediately
git checkout <name> Switch branches (Legacy syntax)
git checkout -b <name> Create and switch to new branch (Legacy syntax)
git branch -d <name> Safe delete a merged branch
git branch -D <name> Force delete an unmerged branch

8. Industry Best Practices

  • Create a dedicated, isolated branch for every distinct feature or bug fix.
  • Keep the main branch protected, tested, and production-ready at all times.
  • Use descriptive, structured branch naming conventions (e.g., feature-login, bugfix-header, hotfix-auth).
  • Commit incremental, logical changes frequently while working within your isolated branch.
  • Delete feature branches promptly after merging to keep the repository's branch index clean.

Summary

In this lesson, you learned what branches are and why they are essential for modern software development. You explored how to create, list, switch, and delete branches using commands like git branch, git switch, and git checkout. Branching enables safe, isolated development, making collaboration and feature development much more efficient.
