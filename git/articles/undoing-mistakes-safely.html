Undoing Mistakes Safely

Learning Objectives

  • Differentiate between destructive and non-destructive undo operations.
  • Unstage staged files safely using git restore --staged.
  • Discard local modifications in the working directory using git restore.
  • Amend the most recent local commit using git commit --amend.
  • Safely undo public shared commits without destroying repository history using git revert.

1. Introduction

Mistakes are inevitable during software development. You may stage the wrong file, modify code incorrectly, or publish a commit containing a bug. Git provides several commands to undo changes safely without losing important work. In this lesson, you'll learn the difference between destructive and non-destructive undo operations and how to use them effectively.

2. Destructive vs. Non-Destructive Operations

Understanding the impact of undo commands is critical to avoiding accidental data loss in Git:

Operation Type Characteristics & Risk Examples
Non-Destructive Preserves project history and timeline. Safe to use on shared public repositories. git restore --staged, git revert, git commit --amend
Destructive Permanently discards uncommitted changes or rewrites history. Use with extreme caution! git restore <file>, git reset --hard

3. Unstaging & Discarding Changes

Unstaging Staged Files

If you accidentally staged a file using git add, move it back to the modified state without altering its contents:

git restore --staged filename

Discarding Local Modifications

Discard uncommitted local edits made to a tracked file and restore it to the last committed version:

git restore filename

Warning: Discarding local edits permanently removes uncommitted code from your disk space!

4. Amending the Last Commit

If you forgot to include a file or made a typo in your commit message, amend your most recent commit instead of creating a brand new one:

git commit --amend

To update only the commit message without modifying code:

git commit --amend -m "Updated commit message"

Note: Only amend commits that are local and have not yet been pushed to a remote server.

5. Reverting Public Commits Safely

When a commit containing an issue has already been pushed to a shared remote repository, do not delete it from history. Use git revert to reverse the changes safely.

git revert <commit-hash>

Git creates a brand new inverse commit that cancels out the changes introduced by the target commit while preserving full historical transparency:

Commit A → Commit B → Commit C
↓ (git revert Commit C)
Commit D (Inverse commit cancelling C)
↓
Safe Shared History Maintained

6. Choosing the Right Command

Scenario / Goal Recommended Command
Remove a file from staging area git restore --staged <file>
Discard local modifications in working tree git restore <file>
Modify code/message of the latest local commit git commit --amend
Undo a commit that has been pushed to a remote repository git revert <commit>

7. Key Commands Reference

Command Purpose
git restore --staged <file> Unstage a file without losing local edits
git restore <file> Discard uncommitted local edits and revert file state
git commit --amend Modify staged files/message of the latest commit
git commit --amend -m "message" Quickly update the message of the latest commit
git revert <commit> Safely create an inverse commit to undo public changes

8. Industry Best Practices

  • Use git restore --staged to fix staging mistakes without touching code edits.
  • Avoid discarding working changes unless you are 100% certain they are no longer needed.
  • Use git commit --amend strictly for local commits that have not been pushed upstream.
  • Use git revert for commits that have already been published to team repositories.
  • Always check git status before and after performing undo operations to verify your working tree.

Summary

In this lesson, you learned how to safely undo mistakes using Git. You explored how to unstage files with git restore --staged, discard unwanted changes using git restore, modify the latest commit with git commit --amend, and safely reverse published commits using git revert. Understanding these commands helps you recover from mistakes while maintaining a reliable project history.
