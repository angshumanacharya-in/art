Git Hooks & Submodules

Learning Objectives

  • Understand how Git Hooks automate client-side and server-side workflows.
  • Configure, write, and execute custom pre-commit and pre-push scripts.
  • Identify common Git hook triggers and their storage location.
  • Understand Git Submodules and how they manage nested repositories with separate histories.
  • Add, clone, update, and manage submodule dependencies across projects.

1. Introduction

As projects grow, automation and code reuse become increasingly important. Git provides two powerful features to help with this:

  • Git Hooks: Automatically execute scripts when certain Git events occur.
  • Git Submodules: Include one Git repository inside another while keeping their histories separate.

These features are widely used in professional software development to automate workflows and manage external dependencies.

2. What are Git Hooks?

Git Hooks are custom executable scripts that Git automatically runs before or after specific lifecycle events, such as committing, merging, or pushing code.

Trigger Event (e.g., git commit)
↓
Execute Pre-Commit Hook Script
↓
Pass Checks? → Proceed or Abort

Common Git Hooks

Hook Name Trigger Point
pre-commit Runs before a commit is created (ideal for linting and code formatting checks).
commit-msg Runs before saving the commit message (used to enforce commit message format rules).
pre-push Runs before code is pushed to a remote repository (ideal for running unit tests).
post-merge Runs after a successful merge operation completes.
post-checkout Runs after switching branches or checking out files.

Hooks are stored locally in your repository at:

.git/hooks/

3. Creating a Simple Pre-Commit Hook

Navigate to the hidden hooks directory and create or edit a hook script:

cd .git/hooks

Add script instructions to the pre-commit file:

#!/bin/sh
echo "Running pre-commit checks..."

Make the script file executable:

chmod +x pre-commit

Every subsequent git commit will execute this script prior to finalizing the commit object.

Benefits of Git Hooks

  • Enforce organizational coding standards automatically.
  • Run automated unit tests before code reaches remote branches.
  • Check and apply uniform code formatting rules.
  • Prevent accidental commits of secrets, API keys, or broken builds.

4. What are Git Submodules?

A Git Submodule allows you to keep a Git repository as a subdirectory of another Git repository, allowing you to clone another project into your repository while keeping your commits separate.

Main Parent Project
↓
Submodule Folder (Nested External Repo)
↓
Independent Commit History Maintained

5. Working with Submodules

Adding a Submodule

To attach an external repository as a submodule to your project:

git submodule add https://github.com/example/library.git

Git creates a .gitmodules configuration file and records the specific commit hash of the external library.

Cloning a Repository with Submodules

To clone a project along with all of its nested submodules in one command:

git clone --recurse-submodules https://github.com/example/project.git

If you already cloned the parent repository without submodules, initialize and fetch them using:

git submodule update --init --recursive

Updating Submodules

Fetch and merge the latest remote commits across all tracked submodules:

git submodule update --remote

6. Key Commands Reference

Command Purpose
chmod +x .git/hooks/pre-commit Make a local hook script executable
git submodule add <url> Include an external repository as a nested submodule
git clone --recurse-submodules <url> Clone a parent repository and all contained submodules
git submodule update --init --recursive Initialize and fetch submodules for an existing clone
git submodule update --remote Fetch and apply the latest remote changes for submodules

7. Industry Best Practices

  • Keep hook scripts lightweight and fast so they do not slow down daily developer workflows.
  • Use repository-managed hook frameworks (like Husky), as local .git/hooks/ files are not committed or shared via Git by default.
  • Use submodules only when projects genuinely require distinct versioning timelines and independent maintenance.
  • Pin submodules to stable tagged commits rather than moving targets like HEAD.
  • Document any required hook setups or submodule setup instructions in your project's README file.

Together, these features make Git workflows more efficient, automated, and scalable for professional software development.
